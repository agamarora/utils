## The Feeling (locked)

"What 2 years of building with AI looks like."

Someone opens their terminal. They see something that has been growing since the day they started using Claude Code. It's visibly complex, visibly personal, visibly the result of time invested. They screenshot it. They share it. Other people look at it and understand immediately: this person has been BUILDING.

It's not entertainment. It's not a game. It's a **credential**. Proof of investment in AI-native development. The thing you show people to say "I've been at this."

## Why This Matters (locked)

Developers who build with AI daily are investing something real — time, attention, craft. There's no way to show that investment today. GitHub contribution graphs show commits. That's the wrong signal now. The new signal is: how deeply, how consistently, how skillfully are you working WITH AI?

Luna-tt makes that investment visible. The social mechanic is: people share what their artifact has become. The uniqueness IS the sharing mechanism. "Look what mine looks like" only works if mine looks different from yours.

## Core Principles (locked)

1. **Watching, not playing.** The use case is 5-15 minute waits while Claude Code runs. Attention is peripheral. It lives in your vision, not your hands.

2. **Growth, not neglect.** It grows because you work, not because you tend it. No care cycle. No hunger bars. No dying. It accumulates.

3. **Procedural generation, not scripted rules.** "Heavy Opus use = ornate features" is a lookup table. Two heavy Opus users end up looking similar. That's not unique, that's a category. The usage data SEQUENCE over time is the seed. The algorithm's output space must be vast enough that no two are alike.

4. **Truthful accumulation.** Can't be faked. Can't be rushed. A 2-year artifact is visibly, obviously different from a 2-week artifact. Time is the ingredient that can't be shortcut.

5. **Multi-year durability.** Not interesting for days or weeks. Interesting for years. The things that last years are honest records of time passing — bonsai trees, sourdough starters, chess ratings, journals. They're interesting because they're unambiguously YOURS in a way that can't be transferred.

## What We Ruled Out (and why)

| Concept | Why it failed |
|---|---|
| Tamagotchi / terminal pet | Neglect/care cycle is wrong for continuous real-time data. Tamagotchis evolved for hourly check-ins. Our data arrives every 30 seconds. |
| Interactive game / idle clicker | Use case is watching/peripheral, not playing. Games also dry out — mechanics get solved. |
| Scripted uniqueness | "Opus = ornate, Sonnet = sleek" is a lookup table. Not truly unique. Two similar users get similar output. |
| Creature as form | Constrains imagination. The level of detail needed for a creature limits the design space. |
| Living World / ambient landscape | Fails uniqueness test. All forests look similar. No reason to come back. |
| Idle RPG / quest log | Narrative text is ephemeral. You don't miss a story the way you miss something that IS you. |
| Entertainment as core value | Wrong frame entirely. This is a credential, not a toy. |

## Data Source (locked)

Luna-proxy (embedded in luna-monitor) is a transparent HTTP proxy for the Anthropic API. It captures response headers and can inspect response bodies.

**Already captured:** 5h/7d utilization, reset times, rate limit status.

**Available but not yet captured (15 headers):**
- Per-minute rate limits: requests, tokens, input tokens, output tokens (limit/remaining/reset for each)
- retry-after, request-id, x-should-retry

**Available from response body (not yet inspected):**
- model (actual model used), stop_reason, usage.input_tokens, usage.output_tokens
- cache_creation_input_tokens, cache_read_input_tokens
- content block types (text, tool_use, thinking)

The data is a multidimensional time series: tokens, models, cache hits, rate limit events, stop reasons, session patterns. The ORDER matters. The RHYTHM matters. This is the seed.

## Terminal Layout (locked)

```
┌──────────────────────────────────────┐
│                                      │
│         LUNA-TT                      │
│         (your history)               │
│         the full journey             │
│                                      │
│         ~50% of vertical space       │
│                                      │
├──────────────────────────────────────┤
│                                      │
│         LUNA-MONITOR                 │
│         (your present)               │
│         CPU, GPU, tokens, rate       │
│         limits, RIGHT NOW            │
│                                      │
└──────────────────────────────────────┘

Width: 20-40% of screen width
Height: full terminal height
```

Top half = history. Bottom half = present. The duality is the design.

## Share Mechanic (locked in concept)

`luna snapshot` — generates a shareable artifact. The social unit. What people post.

The snapshot must be:
- Beautiful enough that non-developers say "what's that?"
- Visually distinct enough that two people's snapshots are obviously different
- Readable enough that someone can tell: this person has invested a LOT vs. a little

## The Open Question: Form

What does the artifact actually LOOK like?

### Candidate A: Organic Growth Structure

Not a geometric mandala. Not a creature. A biological, asymmetric growth pattern — like a tree cross-section, lichen, coral colony. Grows from center outward. Center = day 1. Edge = now.

Properties:
- Bulges where you had intense periods
- Thins where you were quiet
- Branches where you tried new things
- Dense knots where you hit rate limits
- Each layer's texture derived from that period's data

### Candidate B: Constellation / Star Field

Each session = a star. Brightness = tokens. Color = model. Clusters emerge over time. "My sky" is poetic. But: stars don't encode time order inherently.

### Candidate C: Terrain / Topography

Usage data mapped to elevation. Mountains = intense periods. Valleys = quiet. Extends rightward with time.

### Candidate D: TBD — something not yet imagined

The form needs to be SEEN, not described. Visual prototyping is the next step.

### Technical canvas

Unicode braille characters give 2x4 sub-cell resolution per terminal cell. In a 60x20 space that's effectively 120x80 pixels — more than a Game Boy screen. Enough for beautiful forms with true-color support.

The growth doesn't just get bigger — it gets more DETAILED. Early stages are simple (less data to seed the generator). As months pass, textures become richer, patterns more intricate. A 2-year growth is higher fidelity than a 2-month growth, not just larger.

## The Discussion Arc (for context)

1. Started with "terminal tamagotchi" idea
2. Explored: creature, living world, idle RPG, ambient landscape
3. User pushed: "the uniqueness is scripted — we need procedural generation"
4. User pushed: "creature limits our ability to think and imagine"
5. Reframed from entertainment → credential ("what 2 years of building with AI looks like")
6. User: "people will use this because it lets them showcase what they are investing in"
7. Landed: the concept is locked, the form is the open question
8. Best hypothesis: organic growth structure (biological, not geometric)

## Next Steps

1. **Visual prototyping** — render form candidates in actual terminals to see which feels true
2. **Data model** — define exactly which signals feed the generator and how they're stored
3. **Procedural generation research** — L-systems, diffusion-limited aggregation, cellular automata, reaction-diffusion
4. **Proof of concept** — render ONE frame of the strongest form candidate
5. **Capture the missing proxy data** — start logging the 15 uncaptured headers + response body fields

## Name
