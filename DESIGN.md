## Problem Statement

Claude Code users have no unified peripheral awareness of their machine state and Claude usage limits. System monitoring tools (btop, htop, Glances) exist. Claude usage monitoring exists (Claude Pulse as a status bar). Nothing combines them into a full-screen terminal dashboard that shows the complete picture in one glance.

The developer running Claude Code all day needs to know: How much session/weekly usage do I have left? How full is my context window? What model am I running? Is my CPU pegged from a build Claude kicked off? Is my GPU busy? All of this belongs in one view because these are correlated signals, not separate concerns.

## What Makes This Cool

**The fuel gauge for AI cognition.** The headline feature is a usage burndown waveform, a time-series chart (same visual style as the CPU waveform) that tracks Claude usage consumption over time and projects when you'll hit your limit. Panel title updates dynamically: "~47 min of Opus remaining." Nobody has this in a terminal.

The screenshot sells it: a beautifully dense terminal dashboard that looks like a dev mission control. Rich colors, clean panels, live data. The kind of thing people post on Reddit and immediately want to install.

## Constraints

- Windows-first for v1 (user's platform, can dogfood). Architecture isolates OS-specific code for future cross-platform support.
- Claude usage is the primary value. System monitoring is supporting context.
- pip-installable package. Dependencies: psutil, rich, pynvml (optional for NVIDIA GPU). stdlib urllib for HTTP (no requests dependency).
- Claude data from Anthropic OAuth API (same as Claude Pulse: reads token from ~/.claude/.credentials.json, hits api.anthropic.com/api/oauth/usage).

## Premises

1. Full-screen Rich terminal dashboard in its own terminal window/tab (not a status line).
2. Windows-first for v1, architect for cross-platform. psutil works everywhere, OS-specific bits (PDH, IOCTL, LHM) behind platform abstraction.
3. Claude usage is the soul, not a bonus. If the API breaks, the tool shows a clear message. System panels continue but the tool loses its primary reason to exist.
4. pip-installable package with clean pyproject.toml and entry point.

## Approaches Considered

### Approach A: Fork and Extend (single file)
Copy monitor.py, add Claude panels, keep single-file architecture. Fastest to ship but will get unwieldy past ~1000 lines. OS-specific code mixed with portable code. Hard to test. Effort: S. Risk: Low.

### Approach B: Modular Cockpit (chosen)
New package with separate modules: collectors (system, claude, process), renderers (panels), compositor. Each panel is its own file. OS-specific code behind platform abstractions. Config file for panel selection. Entry point via pyproject.toml. Effort: M. Risk: Low-Med.

### Approach C: Panel Plugin System
Full plugin framework with entry_points for third-party panels. Maximum extensibility but premature for v1... nobody writes plugins until the tool has users. Effort: L. Risk: Med-High.

## Recommended Approach

**Approach B: Modular Cockpit.** Balances shipping speed with the architecture needed for open source contributions and cross-platform expansion. Concrete structure:

```
claude-dev-buddy/
  pyproject.toml
  src/claude_dev_buddy/
    __init__.py
    __main__.py           # entry point
    config.py             # user config (TOML or JSON)
    app.py                # Rich Live loop + compositor
    collectors/
      __init__.py
      system.py           # CPU, memory, disk, network (psutil)
      gpu.py              # NVIDIA GPU (pynvml, optional)
      claude.py           # OAuth API, session detection, usage data
      platform_win.py     # PDH disk active time, IOCTL drive mapping, LHM temps
      platform_posix.py   # placeholder for Linux/Mac implementations
    panels/
      __init__.py
      cpu.py              # CPU waveform panel
      memory.py           # RAM + swap panel
      gpu.py              # GPU utilization panel
      disks.py            # Disk active time + I/O panel
      network.py          # Network speeds panel
      temps.py            # Temperature panel
      processes.py        # Top processes (highlight Claude processes)
      claude_status.py    # Active model, context window, session/weekly usage
      claude_burndown.py  # Usage burndown waveform with prediction
    ui/
      __init__.py
      charts.py           # wave_chart(), hbar(), shared rendering utilities
      colors.py           # color thresholds, theming
```

### Core panels (v1):
1. **Claude Status** (top of screen, the soul): Active model, context window fill bar, session usage bar, weekly usage bar, plan tier, time until limit resets
2. **Claude Burndown** (second panel): Time-series waveform of usage consumption with projection line showing estimated time until limit. Dynamic title: "~47 min of Opus remaining"
3. **CPU Waveform**: Rolling area chart (ported from monitor.py)
4. **Memory + GPU**: Side-by-side (ported from monitor.py)
5. **Disks**: Active time + I/O (ported from monitor.py, Windows-only initially)
6. **Network**: Current/avg/peak speeds (ported from monitor.py)
7. **Temps**: CPU + GPU temperatures (ported from monitor.py)
8. **Processes**: Top by CPU/RAM, with Claude-related processes highlighted

### Claude data collection:

**OAuth token flow:**
- Read `~/.claude/.credentials.json` which contains `accessToken`, `refreshToken`, `expiresAt`
- If `accessToken` is expired, refresh via POST to `https://console.anthropic.com/v1/oauth/token` with `grant_type=refresh_token` and the `refreshToken`. Parse the new `access_token` from the response. Store in-memory only (never overwrite the credentials file).
- If refresh fails (revoked token, network error), show "Re-authenticate Claude Code to enable usage tracking" in the Claude panels. Domain allowlist: only send tokens to `api.anthropic.com` and `console.anthropic.com` (hardcoded, same as Claude Pulse).

**Usage API:**
- GET `https://api.anthropic.com/api/oauth/usage` with header `Authorization: Bearer {token}` and `anthropic-beta: oauth-2025-04-20`
- Response JSON structure:
  ```json
  {
    "five_hour": {"utilization": 0.12, "resets_at": "2026-03-27T22:00:00Z"},
    "seven_day": {"utilization": 0.45, "resets_at": "2026-04-03T00:00:00Z"},
    "seven_day_opus": {"utilization": 0.30, "resets_at": "..."},
    "seven_day_sonnet": {"utilization": 0.15, "resets_at": "..."},
    "extra_usage": {"utilization": 0.0, "limit_usd": 0, "spent_usd": 0}
  }
  ```
- Cache for 60 seconds (default, configurable in config file). Cache TTL is the only config key for v1.

**Model/context window data:**
Claude Pulse gets model and context window via Claude Code's `statusLine.command` stdin pipe. This dashboard runs in a separate terminal and cannot access that pipe. Two approaches for v1:
- **Primary:** Claude Pulse already writes parsed stdin context to `~/.claude-pulse/stdin_ctx.json` on each refresh. Read this file if it exists (piggyback on Pulse's data). This works when both tools are running.
- **Fallback:** If stdin_ctx.json is unavailable, omit model name and context window from the Claude Status panel. Show usage data only (session %, weekly %, reset timers). These are the most valuable metrics anyway.
- **Future:** Investigate Claude Code session JSONL files at `~/.claude/projects/*/sessions/` for model and token data.

**Process detection:**
- Match processes named `claude` or `node` where command-line args contain `claude` or `@anthropic`
- Cross-reference with `~/.claude/projects/*/sessions/*.jsonl` PID entries if available

### Burndown prediction algorithm:

The burndown panel tracks `five_hour.utilization` over time in a `deque(maxlen=300)`.

**Projection:** Linear regression over the last 10 data points (5 minutes of data at 30s cache intervals). Extrapolate to 100% utilization to estimate time until limit hit. Display as dim dashed projection line on the waveform chart.

**Cold start:** For the first 5 minutes (fewer than 10 data points), show the waveform without a projection line. Panel title shows "Collecting data..." instead of a time estimate. After 10 points, switch to the dynamic "~X min remaining" title.

**Edge cases:** If usage is flat (slope ~0), show "Pace: sustainable" instead of a time estimate. If usage is decreasing (session reset happened), reset the history deque.

### Panel layout / compositor:

Panels stack vertically in the Rich `Group()` (same approach as current monitor.py). The compositor in `app.py` calls each panel's `build()` function and assembles the Group. Panel widths auto-expand to terminal width. Side-by-side panels (Memory + GPU) use Rich `Table` with column ratios, same as current monitor.py. No grid system needed for v1.

At narrow terminals (<80 columns), side-by-side panels collapse to stacked. The CPU waveform and burndown waveform auto-size their width to `console.width - 4` (same as current monitor.py).

### Config:

Config file at `~/.config/claude-dev-buddy/config.json` (or `%APPDATA%/claude-dev-buddy/config.json` on Windows). For v1, the only keys are:
```json
{
  "cache_ttl_seconds": 60,
  "refresh_seconds": 2.0,
  "drives": ["C:\\", "D:\\"]
}
```
All other settings use sensible defaults. Ship one good theme in v1. Theming is a v2 feature.

## Open Questions

1. **Name**: Is "claude-dev-buddy" the right name? Other options: claude-cockpit, claude-dash, devpulse, codepulse. The name matters for pip and community identity.
2. **OAuth API stability**: This is an undocumented Anthropic API. It could change. The collector should version-tag the expected response schema and fail loudly (with a "please update claude-dev-buddy" message) if the schema changes, rather than silently showing wrong data.

## Success Criteria

- `pip install claude-dev-buddy && claude-dev-buddy` works on Windows with Python 3.10+
- Shows Claude usage (session + weekly) with burndown prediction in <5 seconds after launch
- Shows system metrics (CPU, memory, GPU, disk, network) refreshing every 2 seconds
- The screenshot in the README makes people want to install it
- Graceful degradation: if no NVIDIA GPU, skip GPU panel. If Claude API unavailable, show status message in Claude panels while system panels continue.

## Distribution Plan

- PyPI package: `pip install claude-dev-buddy`
- GitHub Releases for tagged versions
- Entry point in pyproject.toml: `[project.scripts] claude-dev-buddy = "claude_dev_buddy.__main__:main"`
- CI/CD: GitHub Actions for linting, testing, and publishing to PyPI on tagged releases
- README with screenshot is the primary marketing channel
